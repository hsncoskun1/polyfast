## 3. Tasarim Kararlari

### Migration'lar kucuk ve tek amacli
- 003: yalnizca discovery_checkpoints (id, scan_data_json, total_scanned, total_matched, parse_failures, success, scanned_at)
- 004: yalnizca balance_snapshots (id, total_balance, available_balance, fetched_at, session_id, created_at)
- "Ileride lazim olur" diye ekstra kolon EKLENMEDI

### Discovery Checkpoint — tam olarak ne sakliyor
- total_scanned, total_matched, parse_failures: scan istatistikleri
- success: scan basarili mi
- scan_data_json: matched_condition_ids listesi
- scanned_at: scan zamani
- Ownership: persistence katmani saklar, discovery engine uretir

### Balance Snapshot — tam olarak ne sakliyor
- total_balance, available_balance: USD degerleri
- fetched_at: API'den cekilme zamani
- session_id: opsiyonel session baglantisi
- Accounting contract ile iliski: BalanceSnapshot (v0.2.1) ile ayni verileri persist ediyor, session accounting'e destek

### Registry Persistence
- Bu surumde EKLENMEDI
- Neden: Registry su an in-memory ve safe sync her scan'de yeniden olusturuyor. Faz 2'de registry persistence zorunlu degil. Faz 7 recovery surumunde registry state'in persist edilmesi daha dogru timing.

### Persisted State vs Runtime Authoritative State
- DB destekleyici katman — authoritative state discovery/registry/validation tarafinda
- Persistence domain logic URETMIYOR
- checkpoint_store ve balance_store hicbir domain modulu import ETMIYOR
- Save/load contract'lari sade: dict in, dict out

### Empty State Davranisi
- Empty DB'den load → None doner (checkpoint ve balance ikisi de)
- Test dogruladi

### Secret Guvenligi
- Checkpoint ve balance tablolarinda credential alani YOK
- Hassas veri DB'ye plaintext YAZILMIYOR
- Test fixture'larinda gercek secret benzeri deger KULLANILMADI
